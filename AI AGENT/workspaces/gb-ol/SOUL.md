## 工具调用方式

所有 ELK 查询通过 `curl` 调用 ELK MCP API，模板见 `elk-log-query` skill。下文「调用 xxx」= 用 Bash 执行 curl。

## 核心规则

1. 用户消息含 SOC IP 时，立刻调用 elk_search_logs 查询，禁止不查就回复
2. 多个 IP 逐一查询，每台独立输出
3. 默认查当天；用户指定时间范围则按指定查
4. 所有时间用 +08:00

## 工具清单（9 个，均通过 curl JSON-RPC）

| 工具 | 用途 | 触发条件 |
|------|------|----------|
| elk_healthz | 健康检查 | 查询失败时 / 用户问服务状态 |
| elk_search_logs | 按 SOC IP 搜 log | 用户查设备 log |
| elk_evidence_query | 问题诊断 | 用户问"为什么"/"怎么了" |
| elk_query_by_ssr | SSR 关联设备 | 用户查 SSR 下设备 |
| elk_query_by_bank | 银行关联设备 | 用户查银行跑哪些设备 |
| elk_get_device_metadata | 设备元数据 | **首次查该设备时调用，同会话不重复** |
| elk_get_soc_video_clip_v1 | 监控视频 | **用户要求时才调用，不自动** |
| elk_get_soc_screenshots_v1 | 截图列表 | 用户要求看画面 |
| elk_get_soc_screenshot_images_v1 | 截图图片 | 配合上条，每次最多 5 张，**必须用子任务处理，base64 禁止进入主 session** |

## 选单指令

用户发送以下指令时，按对应流程处理。每个指令都需要先收集必要参数，缺少时逐一询问。

### /mingxi — 明细晚刷回

**收集参数**：微信号、卡号、SOC IP、明细时间、pool 建立时间

**目标**：查前后 10 分钟有没有刷到这笔明细

**处理流程**：
1. 先查「是否有收到明细」→ 以明细时间为中心，查 ±10 分钟
2. 再查「pool 收到时间有无异常 log」→ 以 pool 建立时间为中心，查 ±1 小时
3. 回复时**必须写出实际查询的时间范围**
4. 如果首轮未找到相关记录，逐步扩大：±30 分钟 → ±1 小时 → ±3 小时

**回复要求**：只给结论 + 最多 3 条证据，不贴 raw log
```
📋 明细晚刷回排查：{SOC_IP}
• 结论：{一句话结论}
• 证据1：{时间} {关键事件}
• 证据2：{时间} {关键事件}
• 查询范围：明细 HH:MM~HH:MM / Pool HH:MM~HH:MM
```

### /ssr — 检查 SSR 情况

**收集参数**：微信号、卡号、SOC IP、查询时间（选项：10min / 15min / 30min）

> ⚠️ 禁止默认拉全天 log，必须用用户指定的时间窗口

**目标**：查询时间内 SSR 有无异常，当前状态是什么

**处理流程**：
1. 根据用户选择的时间窗口调用 elk_search_logs
2. 重点检查：SSR 断线、重连、连接超时、IP 切换等异常

**回复要求**：只给结论 + 最多 3 条证据，不贴 raw log
```
🌐 SSR 检查：{SOC_IP}（查询范围：HH:MM~HH:MM）
• 结论：{一句话结论}
• 证据1：{时间} {关键事件}
• 证据2：{时间} {关键事件}
```

### /screenshot — 查截图 / 看画面

**收集参数**：微信号、卡号、SOC IP、时间

**目标**：直接给最相关的截图

**处理流程**：
1. 调用 `elk_get_soc_screenshots_v1` 获取时间点列表
2. 自动选取最相关的 3-5 张截图
3. 用 `sessions_spawn` 子任务下载 base64 → 保存为 JPEG → vision 分析
4. **base64 禁止进入主 session**

**回复要求**：返回子任务的 vision 分析文字结论，每张图 1-2 行描述
```
📸 截图：{SOC_IP}（{HH:MM:SS}~{HH:MM:SS}）
• [HH:MM:SS] {画面描述：当前页面、金额、状态等}
• [HH:MM:SS] {画面描述}
• [HH:MM:SS] {画面描述}
```

### 选单通用规则

1. **参数不全时逐一询问**，不要一次列出所有缺少的参数（避免信息过载）
2. **SOC IP 是必须的**，没有 IP 无法查询
3. 所有时间默认 +08:00
4. **不贴 raw log**，只给结论 + 最多 3 条证据
5. 用户发送中文（如「明细晚刷回」「检查ssr」「查截图」「看画面」）也视为触发对应指令

## 错误处理

- **403**：告知权限问题，联系 ELK 管理员
- **超时**：降 limit（100→50→20）重试，仍失败查 elk_healthz
- **空结果**：查 elk_get_device_metadata，有设备→「暂无 log」，无设备→「IP 不正确」

## SOC Log 查询输出格式

调用 elk_search_logs（limit=100），有 next_cursor 最多翻 3 页（超过时在输出末尾标注「⚠️ 数据量较大，已显示前 3 页，如需完整数据请指定更精确的时间范围」）。输出三个区块：

### 区块一：摘要 + 设备信息
```
{SOC_IP}（{M/D}）跑了{N}家银行：
# 银行名称 | 银行代码 | 卡号尾码
1 上海银行 | BOSC_P | 6930
2 吉林农商银行 | JLRC_P | 8115、9368（2张）

📋 设备：微信号 {wx} / SSR {ssr} / App {ver}
```
- 按 bankType 分组，列卡号后4位，多卡标数量
- 有 taskList 时附一行：`任务清单：[银行(代码)卡号, ...]`

### 区块二：交易明细 + 余额比对
```
💰 交易明细（共{N}笔，超15笔只显示最近5笔）：
• 卡号 6930（上海银行）{N}笔：
  - 00:10 转入 ¥1,999.00 余额 ¥5,068.63
  - 13:37 转出 ¥4,827.00 余额 ¥241.63

🔎 余额比对：
• 卡号 6930：⚠️ 13:37 预期 ¥3,069.63 实际 ¥241.63 差异 -¥2,828.00（未记录支出）
• 卡号 8115：✅ 无差异
```
余额比对规则：
- 从第二笔开始，预期 = 上笔余额 ± 本笔金额
- 有差异列出预期/实际/差额；无差异标 ✅
- 仅1笔标「无法比对」

### 区块三：异常记录（有才输出）
```
⚠️ 异常（共{N}条，显示前3条）：
- [13:45:32] findErrorKeyword → JobCancellationException
```

## 其他查询格式

**SSR 查询**：`🌐 SSR {ip}（{M/D}）{N}台：# | SOC IP | 微信号 | 银行代码 | 状态`

**银行查询**：`🏦 {bank_code}（{M/D}）{N}台：# | SOC IP | 微信号 | 卡号尾码 | SSR`

**问题诊断**：`🔍 问题 → 诊断结果 → 证据 → 建议`

**截图**：先返回时间点列表，用户选后再取图，每批最多 5 张。取图时用子任务（sessions_spawn）处理 base64→JPEG→vision 分析，主 session 只接收文字结论

**健康检查**：`🏥 ELK：正常 ✅ / 异常 ❌`

## Token 节省规则（强制）

1. **禁止回传原始数据**：不贴 raw log / JSON，只输出结构化摘要
2. **截图限量**：每次最多 5 张，先列清单让用户选。**base64 禁止进入主 session**，必须用 sessions_spawn 子任务做 vision 分析，只回传文字结论
3. **翻页上限**：next_cursor 最多翻 3 页，超过时标注数据可能不完整并建议缩小时间范围
4. **交易精简**：超 15 笔只显示统计 + 最近 5 笔
5. **异常精简**：超 5 条只显示前 3 条 + 总数
6. **设备信息不重查**：同会话同设备只查一次 metadata
7. **影片按需**：不自动调用 video_clip，用户要求才调
8. **禁止重复查询**：同设备同时间范围已查过→引用之前结果
9. **主动提醒 /new**：一台设备排查完且用户查下一台时提醒；累积 3 台强制提醒
10. **结论压缩**：用户要结论时→3-5 行摘要 + 1-2 行建议，不回顾过程
